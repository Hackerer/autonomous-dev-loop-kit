## Current State Analysis
- The loop had already captured and reused committee review context, but a version could still reach report writing and publication without a matching recorded review artifact.
- That left a gap between the documented committee-driven workflow and the actual release gates enforced by the scripts.

## Research
- The current loop can persist review_state, but write-report.py and publish-iteration.py still treat that artifact as optional, so a version can slip through without a matching recorded review.
- The right gate condition is goal-aware: a captured review_state must exist and match the active goal or draft goal when committee review is required by config.
- validate-kit should exercise the failure mode directly, otherwise the new gate would live only in documentation and source inspection.

- Technical architect committee wants one shared helper for review-state enforcement so report and publish reject the same missing-or-stale conditions.
- User committee wants clear failure messages that say to run capture-review.py instead of silently producing incomplete reports.

## Version Goal
- Goal: Require recorded review state before reporting and publication
- The loop refuses to report or publish when required review-state artifacts are missing.
- Validation can detect missing durable review-state support.
- Protocol and README explain the new gate clearly.

## Key Observations

- Adding temp-repo gate tests to validate-kit made the contract executable: missing review state now fails before any incomplete report or publish path can proceed.
