## Scope Decision
- Why selected: This is the most severe open finding because it can corrupt release history and session progress at publish time.
- Scope in: Preflight remote requirements before mutating iteration or release state.
- Scope in: Commit successful publish state inside the published artifact instead of leaving runtime state dirty afterward.
- Scope in: Add validator checks for clean post-publish state and failed-publish non-consumption.
- Scope out: Do not change release planning semantics or committee structure.
- Assumption: Recording the final publish state in the committed artifact matters more than preserving commit SHA inside state.
- Risk: Publish event payloads may need to drop the commit SHA from pre-commit usage logging to keep the transaction clean.
- Required validation: validate-kit.py must prove successful release publish leaves a clean worktree.
- Required validation: validate-kit.py must prove failed iteration publish leaves goal and backlog state untouched.
- Stop condition: Stop if the fix requires destructive rollback of user-authored commits.
- Scope dissent: Do not introduce heavy transactional machinery that changes the loop protocol just to preserve a self-referential commit field.
- Next action: Run validate-kit.py and publish the remediation release if the transactional ordering remains green.
