- Research gate summary: Goal selection should stop only when durable project-data quality or the structured research gate explicitly says more context is required.

## Scope Decision
- Why selected: This closes a protocol gap: the repo already modeled research gating, but goal selection could still continue without respecting insufficient-context signals.
- Scope in: Add an explicit research gate status for need_more_context.
- Scope in: Block goal selection when quality status is insufficient.
- Scope in: Block goal selection when research_gate.status is need_more_context.
- Scope in: Echo blocking gaps clearly in the error path.
- Scope in: Cover both paths in validate-kit.
- Scope out: Do not require research capture before every goal selection in this version.
- Scope out: Do not add a new workflow engine or transcript storage.
- Scope out: Do not redesign data-quality scoring thresholds.
- Assumption: Existing quality status values remain ready, usable-with-gaps, and insufficient.
- Assumption: A missing quality file should not crash goal selection if a last-known status already exists in state.
- Risk: Over-blocking could make lightweight flows too heavy if usable-with-gaps is treated like insufficient.
- Risk: A stale quality path could produce confusing failures unless the loader is tolerant.
- Required validation: python3 .agent-loop/scripts/validate-kit.py
- Stop condition: Stop if the new blocker prevents normal ready flows from selecting a goal.
- Stop condition: Stop if the change requires broader workflow rewrites outside capture-review and select-next-goal.
- Scope dissent: Keep the gate keyed to explicit insufficient-context signals only; do not infer blockers from every open gap yet.
- Next action: Validate the new blocker end to end, then document the new need_more_context path in the protocol and example workflow.
